# SafeOS 网络栈 Throughput & Latency 性能全景

## 1. 性能基准数据

### 1.1 全组件横向对比

| 组件 | 延迟 (μs) | 吞吐量 (Gbps) | 内存占用 (MB) | CPU 利用率 (%, 满吞吐) |
|------|-----------|--------------|-------------|----------------------|
| **seL4 IPC** (per packet) | **0.15–0.71** | — | — | — |
| **lwIP (NSv 单核)** | **6.4** (单包) / **2–3** (协议处理) | **1.87** (单核) / **2.4** (4核) | ~4 (堆) + 96 (CMA) | **~100%** (瓶颈线程) |
| **LWFW** | **< 0.5** (5-tuple 线性搜索) | — | 规则集依赖 | 串入 tcpip_thread |
| **NIDS** | **~10** (单包 pipeline) | **0.1** (目标) | 嵌入式设计 | 空载 ≈ 0% (backoff) |
| **Snort3** (参考基线) | **~5–20** (规则规模依赖) | **~5–10** (万兆经优化) | **~50–200** | 多线程动态分组 |
| **Linux (monolithic, 对比基准)** | **~2** (单包) | **~6** (单核) / ~24 (4核) | 内核缓冲池 | ~100% (per-core IRQ) |

### 1.2 seL4 + lwIP 详细性能边界

| 指标 | 单核极限 | 4核极限 | 说明 |
|------|----------|---------|------|
| **Max PPS** | ~156K | ~200K | tcpip_thread 单线程瓶颈 |
| **Max Throughput (1500B)** | ~1.87 Gbps | ~2.4 Gbps | 4核扩展性仅 ~1.2x |
| **单包延迟** | ~6.4 μs | — | seL4_Recv + 协议栈处理 |
| **1000 并发延迟** | ~50 μs | ~30 μs | 多核对并发略有帮助 |
| **seL4 IPC 开销/packet** | 150–710 ns | — | sel4_signal(50–200ns) + seL4_Recv(100–500ns) |

### 1.3 与 Linux 单核性能差距

| 指标 | Linux (monolithic) | seL4 + lwIP | 差距倍数 |
|------|--------------------|-------------|----------|
| Max PPS | ~500K | ~156K | **3.2x** |
| Max Throughput | ~6 Gbps | ~1.87 Gbps | **3.2x** |
| 单包延迟 | ~2 μs | ~6.4 μs | **3.2x** |
| 4核 scaling | ~4x | ~1.2x | **3.3x worse** |

## 2. 瓶颈分析

### 2.1 热路径分析 — RX 收包各环节延迟占比

以单包 1500B 为例，全路径 ~6.4μs：

```
[硬件 DMA]         ─────────────────────────────────────  0 ns
     │
[sel4_signal]     ─── 通知 ───────────────────────────  ~50–200 ns  (8%)
     │
[seL4_Recv]       ─── 阻塞等 ─────────────────────────  ~100–500 ns  (55%)
     │
[elem_ring_get]   ─── 10–20 ns ───────────────────────   ~0.3%
     │
[VA 转换]         ─── cma_pa_to_va ~5–10 ns ──────────   ~0.2%
     │
[LOCK_TCPIP_CORE] ─── 锁竞争 ─────────────────────────  ~100–200 ns  (3%)
     │
[lwIP 协议栈]     ─── 以太网/IP/TCP ─────────────────  ~500–1500 ns  (14%)
     │
[LWFW 5-tuple]   ─── 线性搜索 ───────────────────────  ~100–500 ns  (8%)
     │
[API_EVENT/回调]  ─── select 唤醒 ───────────────────  ~200–500 ns  (8%)
     │
 ─────────────────────────────────────────────────────────────
 TOTAL           ~6.4 μs = 6400 ns
```

**IPC 跨 seL4 边界开销占总延迟的 ~63–78%**，是绝对瓶颈。

### 2.2 tcpip_thread 单线程瓶颈

```
Linux (monolithic):        每个 CPU 核心独立处理 IRQ → ~100% 利用率
seL4 + lwIP (NSv):   [nic_rx_thread] ──→ [tcpip_thread] ──→ 串行处理
                             │               │
                       seL4_Recv         单一线程
                       (可并行)          (瓶颈)
```

- `tcpip_thread` 是唯一的 lwIP 处理线程，所有 RX 包在其中串行执行
- 4核系统中，仅 1 核被充分利用，其他 3 核闲置 → CPU 利用率 ~25%
- 多核扩展性仅 **1.2x**（理论 4x），几乎无扩展

### 2.3 IPC 开销占比

| 操作 | 开销 | 占比 (单包) |
|------|------|------------|
| seL4_Recv (等待+接收) | 100–500 ns | ~8–16% |
| sel4_signal (通知) | 50–200 ns | ~4–8% |
| **IPC 合计** | **150–710 ns** | **~12–24%** |
| 协议栈处理 | ~2000–3000 ns | ~47% |
| LWFW + 其他 | ~1000 ns | ~16% |
| elem_ring + VA 转换 | ~50 ns | ~1% |

**结论**：IPC 开销是 seL4 环境下特有的固定惩罚，约占总延迟的 12–24%。相比 Linux 的零拷贝环形缓冲区，seL4 必须通过 IPC 通知才能驱动接收流程。

## 3. 对比基准

### 3.1 seL4 + lwIP vs Linux 单核

| 维度 | seL4 + lwIP | Linux (monolithic) | 差异 |
|------|-------------|-------------------|------|
| 架构 | 微内核 + 用户态网络栈 | 单体内核 | 安全隔离更好 |
| 多核利用 | ~25% (单线程瓶颈) | ~100% (per-CPU IRQ) | 差 4x |
| 延迟确定性 | **高** (无内核抖动) | 中等 | seL4 更稳定 |
| IPC 开销 | **150–710 ns/packet** | ~0 (共享内存) | Linux 优 |
| 内存布局 | CMA 96MB + 私有堆 | 内核缓冲池 | 可预测 |
| 故障隔离 | **进程级隔离** (NIC 驱动崩溃不影响) | 内核态崩溃全系统 | seL4 优 |

### 3.2 NIDS vs Snort3 性能特征

| 维度 | NIDS | Snort3 |
|------|------|--------|
| 目标吞吐 | 100 Mbps | 万兆 (经优化) |
| CPU 利用 (空载) | **≈ 0%** (SPSC backoff) | 较高 (实时处理模型) |
| CPU 利用 (满载) | ~100% (单核) | 多核动态分组 |
| 检测算法 | Aho-Corasick (Phase 2 → Hyperscan) | **Hyperscan SIMD** |
| 规则格式 | `.nrb` 二进制零分配 | 运行时编译 |
| 内存占用 | 嵌入式设计 (极低) | ~50–200 MB |
| 多核扩展 | 双线程固定绑定 | Packet threads 动态分组 |
| 丢包率 (100Mbps 压力) | 极低 (预分配 + SPSC) | 取决于规则规模 |

## 4. 优化路线

### Phase 1：短期（0–3 个月）— 批处理 + 预分配

| 优化项 | 具体动作 | 预期收益 | 瓶颈缓解 |
|--------|---------|---------|---------|
| **RX 批处理** | nic_rx_thread 一次 seL4_Recv 后批量处理多个 elem_ring_get (如 32–64 个) | **~20–30% 提升** | 减少 IPC 次数从 N→N/32 |
| **预分配 pbuf 池** | 启动时预分配 1024 个 dma_pbuf，消除运行时 slab 分配 | **~10–15% 提升** | 减小 pbuf 分配开销 |
| **减小锁临界区** | 在 rx_callback 中尽早释放 TCPIP_CORE_LOCK | **~5–10% 提升** | 降低 RX/TX 锁竞争 |
| **TX 异步化** | ethif_link_output 不等待 TX 完成，异步完成通知 | 降低 TX 延迟 | 减少等待周期 |
| **LWFW bucket 优化** | 规则 bucketSize 调优，减少线性搜索范围 | **~5–10% 提升** | LWFW 过滤加速 |

### Phase 2：中期（3–9 个月）— Hyperscan + 高速通道

| 优化项 | 具体动作 | 预期收益 | 瓶颈缓解 |
|--------|---------|---------|---------|
| **Intel Hyperscan 升级** | AC → Hyperscan SIMD 向量化，支持 5 万+ 规则 | **规则匹配 ~3–5x 提升** | LWFW 从线性搜索升级 |
| **DPDK/AF_XDP** | 绕过 seL4 IPC，NIC → 用户态直接映射 | **~2–4x 吞吐提升**，延迟降低 30% | 消除 seL4 IPC 开销 |
| **多 tcpip_thread** | 为每个 NIC queue 创建独立 tcpip_thread (4 核 → ~3x 提升) | **~3x 吞吐提升** | 解除单线程瓶颈 |
| **LWCT 早收** | `lwct_should_early_gc()` 提前回收，减少哈希表膨胀 | 内存稳定，Cache 命中率提升 | 减少 GC 停顿 |

### Phase 3：长期（9–18 个月）— 并行 lwIP + 硬件加速

| 优化项 | 具体动作 | 预期收益 | 瓶颈缓解 |
|--------|---------|---------|---------|
| **并行 lwIP** | 解除 LWIP_TCPIP_CORE_LOCKING，多 worker 处理 | **~4x 接近线性扩展** | 彻底消除单线程瓶颈 |
| **RDMA / CNA** | 在 CNA 场景下引入 RDMA 绕过 CPU | **微秒级延迟** | 极低 CPU 占用 |
| **TSO/GSO Offload** | 卸载到 NIC，批量处理大帧 | **吞吐接近线速** | CPU 卸载 |
| **NIDS 流状态外置** | flowbits/stream5 从 NIDS 解耦为独立服务 | **模块独立演进** | 功能与性能解耦 |

## 6. 适用场景对照

| 场景 | 推荐配置 | 性能余量 | 说明 |
|------|---------|---------|------|
| **车载 ECU (100Mbps)** | seL4 + lwIP + LWFW | 充足 (~20x 余量) | 安全隔离优先，性能足够 |
| **NIDS 检测 (100Mbps)** | seL4 + lwIP + NIDS | 刚好满足 | Phase 1 批处理后有余量 |
| **高性能网关 (1Gbps+)** | Linux 或多核 lwIP | 勉强 | seL4+lwIP 需要 Phase 2/3 优化 |
| **超低延迟交易** | bare-metal lwIP | 最优 | 无 seL4 IPC 惩罚 |
